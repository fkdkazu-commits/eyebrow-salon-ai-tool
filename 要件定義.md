# 眉毛サロン比較メディア 店舗情報AI自動更新システム 要件定義書

> 作成日: 2026-07-15
> 作成者: システムアーキテクト / PM（Claude Code）
> 対象システム: Claude Codeで開発するAI自動化ツール（ツールA / ツールA' / ツールB）

---

## 1. 背景

眉毛サロン比較メディアは現在**400記事以上**を保有しており、各記事には複数の店舗情報（住所・営業時間・料金など）が掲載されている。

これらの店舗情報は本来ホットペッパービューティー（以下HPB）の掲載内容を正として管理すべきだが、現状は以下のような状態にある。

なお、店舗情報（住所・料金・営業時間等）はHPB上に掲載されているのが基本だが、店舗によってはHPB側に一部項目（特に料金の詳細やその他情報）が掲載されておらず、各店舗の公式ホームページ（以下、公式HP）にのみ情報が存在するケースがある。本システムでは**HPBを第一情報源（正）とし、HPBに記載がない項目のみ各店舗の公式HPを補完的な情報源として利用する**方針とする。

- 店舗情報が記事ごとに手作業で入力されており、**マスタとなるデータが存在しない**
- 記事の構造（table / タブ / div など）がバラバラで、統一フォーマットになっていない
- HPB側の情報（営業時間・料金・定休日など）が変更されても、記事側に反映されず**情報が古いまま放置される**リスクがある
- 400記事 × 複数店舗という規模のため、**人力でのメンテナンスは事実上不可能**になりつつある

## 2. 目的

1. HPBを正とし、必要に応じて各店舗の公式HPで補完した**店舗マスタ**をGoogleスプレッドシート上に構築し、店舗情報の一元管理を実現する
2. 400記事以上に散在する店舗情報表を、AIの力で**統一フォーマットへ一括置換**する
3. HPB側の情報変更を**定期的に自動検知**し、マスタ更新と変更履歴の記録を自動化する
4. 上記を通じて、店舗情報の**正確性・鮮度を維持しながら運用工数を大幅に削減**する

## 3. 課題（As-Isの問題点）

| # | 課題 | 影響 |
|---|---|---|
| 1 | 店舗マスタが存在しない | 情報の正本がなく、記事間で表記揺れ・情報の食い違いが発生 |
| 2 | 記事の構造が不統一（table / タブ / div等） | 一括置換や機械的な更新ができない |
| 3 | HPBの情報更新を検知する仕組みがない | 古い営業時間・料金・定休日等がユーザーに提示され続ける（信頼性低下、機会損失） |
| 4 | 400記事 × 複数店舗の手動確認は工数過多 | 更新作業が後回しになり、鮮度がさらに悪化する悪循環 |
| 5 | 変更履歴が残らない | いつ・何が・どう変わったかを追跡できず、誤更新の検証ができない |

        │
        ▼
[担当者が気づいたら手動で該当記事を検索]
        │

        │

        │
        ▼

        │
        ▼

**問題点**: 起点となる「変更に気づく」プロセスが存在しないため、実際にはほとんど更新されない。

## 6. システム概要

本プロジェクトでは、以下3つの独立したAI自動化ツールをClaude Code（Node.js / TypeScript）で開発する。

| ツール | 役割 | 実行頻度 | 起動方法 |
|---|---|---|---|
| **ツールA** | HPBを第一情報源として店舗情報を取得し、不足項目は公式HPで補完してGoogleスプレッドシートに店舗マスタを新規構築する | 都度実行（新規店舗追加時） | CLI手動実行 |
| **ツールA'**（店舗情報自動反映ツール） | 既存記事内の店舗情報表を、店舗マスタを参照して統一フォーマットへ置換し、WordPress REST API経由で記事へ自動反映する | 都度実行（初回一括移行＋差分承認後の随時反映） | CLI手動実行（承認済み差分の自動反映はツールBから呼び出し） |
| **ツールB** | HPBを定期巡回し、店舗マスタとの差分を検知・記録する。あわせて、承認済み（approved=TRUE）の差分があればツールA'を自動起動し記事へ反映する | 定期実行（日次 or 週次を想定） | タスクスケジューラ/GitHub Actions（自動） |

3ツールはいずれもCLIベースのNode.js/TypeScriptツールとし、共通のスクレイピング基盤（Playwright）・LLM抽出基盤・Google Sheets連携基盤を共有ライブラリとして再利用する。

## 7. 機能一覧

### 7.1 ツールA（店舗マスタ作成）

| # | 機能 | 概要 |
|---|---|---|
| A-1 | 入力受付 | 店舗URL一覧 or 店舗名一覧（CSV/スプレッドシート）を受け取る |
| A-2 | 店舗名検索 | 店舗名のみ入力された場合、HPB検索経由でURLを特定する |
| A-3 | 店舗情報取得 | Playwrightで店舗ページを開き、基本情報（店舗名/住所/営業時間/定休日/アクセス/電話番号/支払方法/駐車場/口コミ件数/評価/その他）を取得 |
| A-4 | クーポン情報取得 | クーポンページを巡回し、初回料金/通常料金/メンズ料金/学割/人気メニュー等をLLMで判定・抽出 |
| A-5 | 公式HPフォールバック取得 | HPBから取得した項目に欠落（null）がある場合、店舗の公式HP（URLが登録されている場合）を巡回し、該当項目のみをLLMで補完抽出する。**HPBの値がある項目は上書きしない**（HPB優先の原則） |
| A-6 | データ正規化 | 取得した生データをスプレッドシートの列定義に合わせて整形する |
| A-7 | スプレッドシート書き込み | Google Sheets APIで店舗マスタシートへ1店舗1行で書き込み（新規追加・既存更新の両対応）。各項目の情報源（HPB/公式HP）を記録する |
| A-8 | 実行ログ出力 | 取得成功/失敗、スキップ件数、公式HPフォールバック件数などをログシートまたはログファイルへ記録 |
| A-9 | 閉店判定 | 登録対象のHPB/公式HP URLに正常にアクセスでき、かつページ内に「閉店」「サービス終了」「掲載期間が終了しました」等の**明確な閉店を示す文言が含まれる場合のみ**、当該店舗を閉店候補として`status`列に記録する。HTTP 404/403やタイムアウト等の**アクセスエラーは閉店の根拠にしない**（下記A-9-1参照） |
| A-9-1 | アクセスエラーの区別 | HTTP 404/403/タイムアウト・接続失敗等は、Bot対策・URL変更・サイト一時停止など閉店以外の原因が大半のため、`status`は`closed`にせず`error`（要確認）として記録するに留める。`error`が継続する場合も自動的に`closed`へは遷移させず、人手による個別確認を促す |
| A-9-2 | HPB「掲載期間終了」の扱い | HPBの「掲載期間が終了しました」はHPBへの掲載契約が切れただけで、**実店舗が閉店したことを意味しない**（同一店舗が別URLで営業継続しているケースが実務上多数を占める）。このメッセージのみを根拠に`closed`候補とはせず、`status`を`hpb_expired`（要再調査）として記録し、公式HP等で別URLが存在しないか確認するフローに回す。`closed`候補とするのは、公式HP・SNS等**店舗自身の発信**で閉店が明言されている場合に限る |

### 7.2 ツールA'（店舗情報自動反映ツール）

| # | 機能 | 概要 |
|---|---|---|
| A'-1 | 対象記事取得 | WordPress REST APIで更新対象記事一覧・本文HTMLを取得 |
| A'-2 | 構造解析 | AI + HTML解析ライブラリで、記事内の店舗情報部分（table/タブ/div/ショートコード等）を特定 |
| A'-3 | 店舗名寄せ（名寄せ） | 記事内に記載された店舗名・情報から店舗マスタ上の該当店舗をAIで同定する |
| A'-4 | 統一フォーマット生成 | 店舗マスタの最新情報から統一HTMLフォーマットを生成 |
| A'-5 | 部分置換 | 店舗情報部分のみを新フォーマットに置換し、紹介文等の非対象部分は変更しない |
| A'-6 | プレビュー/ドライラン | 本番反映前に差分をプレビューできるドライランモードを提供 |
| A'-7 | WordPress更新 | WordPress REST API経由で記事を更新（下書き保存 or 即時公開を選択可能） |
| A'-8 | 実行結果レポート | 成功/失敗/要確認（AI判定に自信がない）記事の一覧をレポート出力 |
| A'-9 | 差分店舗のみ反映モード | store_id等を指定して実行すると、全記事ではなく該当店舗を含む記事のみを検索・更新する。さらに、ブロック内の全項目を再生成するのではなく、変更履歴シートで承認された**変更項目（field）のみ**をピンポイントで書き換える（変更のない項目・他店舗のブロック・紹介文は一切変更しない。ただしstatus=closedの場合はブロック全体を意図的に置換する。A'-10参照）。変更履歴シートでapproved=TRUEになった店舗をツールBが自動検出し、本モードで呼び出す（詳細は7.3のB-6） |
| A'-10 | 閉店店舗のブロック置換 | 対象店舗の`status`が`closed`（承認済み）の場合、通常の統一フォーマット置換ではなく、当該店舗ブロックを**「閉店しました」の旨を示す短い注記**に置換する。記事の構造（他店舗の並び順・件数）は維持する |

### 7.3 ツールB（差分検知）

| # | 機能 | 概要 |
|---|---|---|
| B-1 | 定期巡回 | 店舗マスタに登録済みの全店舗についてHPBページを再取得 |
| B-2 | 差分比較 | 取得した最新情報と店舗マスタの現在値を項目単位で比較 |
| B-3 | セル色付け | 差分があった項目のセルを黄色にハイライト |
| B-4 | 変更履歴記録 | 変更履歴シートに「日時/店舗/項目/旧値/新値」を追記 |
| B-5 | 実行ログ記録 | 巡回件数、成功/失敗、差分検知件数などをログシートへ記録 |
| B-6 | 承認済み差分の自動反映 | 変更履歴シートの`approved`列がTRUEかつ`reflected_to_articles`がFALSEの行を検出し、該当店舗を対象にツールA'（差分店舗のみ反映モード）を自動起動してWordPress記事へ反映する。WordPress更新を伴うため**完全自動化はせず、担当者が変更履歴シート上で事前に承認（approved列をTRUEに変更）することを必須**とする |
| B-7 | 閉店判定 | 再取得時にページへ正常アクセスできた上で、**店舗自身が発信する**「閉店」「サービス終了」等の明確な閉店文言を検知した場合のみ、店舗マスタの`status`を`closed`候補として更新し、変更履歴シートに`field=status`の行として記録する（他項目と同様、approved承認後にツールA'が記事側を「閉店しました」表示へ置換する）。HPBの「掲載期間が終了しました」は`hpb_expired`として区別し（A-9-2参照）、HTTP 404/403・タイムアウト等のアクセスエラーは`error`として記録するのみで、いずれも`closed`候補には**しない**（実行ログシートに要確認として残す） |

## 8. 非機能要件

| 項目 | 内容 |
|---|---|
| 実行環境 | Windows / Node.js（LTS） / TypeScript |
| 実行方式 | CLI実行（ツールA・A'は手動トリガー、ツールBは定期実行想定。承認済み差分の記事反映はツールBの定期実行時に自動判定） |
| 対象データ量 | 記事約400件、店舗数は記事あたり複数（実店舗数は数百規模を想定） |
| 処理時間 | ツールA: 1店舗あたり数秒〜数十秒（LLM呼び出し込み）。ツールBの全件巡回は深夜バッチ等での長時間実行を許容 |
| 可用性 | 常時稼働は不要。定期実行はタスクスケジューラ/GitHub Actions等の外部スケジューラに依存 |
| スクレイピング耐性 | HPB側のレイアウト変更・アクセス制限（bot対策）に対して、リトライ・待機・User-Agent設定等で耐性を持たせる |
| データ整合性 | Google Sheetsを唯一の正（Single Source of Truth）とし、記事側は常にマスタからの一方向反映とする |
| セキュリティ | APIキー・認証情報はプロジェクトフォルダ外の専用シークレットフォルダで管理し、リポジトリには含めない（[8. ディレクトリ設計](./設計書.md#8-ディレクトリ設計)参照） |
| ログ・監査性 | 全ツールの実行結果をログシートまたはログファイルに記録し、後から追跡可能にする |
| 保守性 | HPBのHTML構造変更に対し、スクレイピング用セレクタ/ロジックを局所修正で追従できる設計とする |
| 拡張性 | 眉毛サロン以外の業種（美容室・ネイルサロン等の比較メディア）へ横展開できるよう、業種固有ロジックを設定ファイル/プロンプトに分離する |

| 項目 | 内容 |
|---|---|
| 開発言語・環境 | Node.js + TypeScript（Claude Codeでの開発を前提） |
| スクレイピング | Playwrightを使用すること |
| 情報抽出 | LLM（Claude Code CLI／Cowork経由）を利用すること。従量課金のAPIキーは使用せず、既存のClaudeサブスクリプションの範囲で実行する |
| スプレッドシート連携 | Google Sheets APIを利用すること |

| バージョン管理 | Gitで管理すること |
| 対象サイト | ホットペッパービューティーの利用規約・robots.txt の範囲内でのアクセスに限定する（過度な高頻度アクセスは行わない） |
| シークレット管理 | APIキー等はプロジェクトフォルダ内に一切残さない（グローバルルールに準拠） |

## 10. 前提条件

- HPBの店舗ページ・クーポンページはログイン不要で閲覧可能な公開ページであることを前提とする
- WordPressサイトはREST APIが有効化されており、記事更新用の認証情報（アプリケーションパスワード）が別途発行されることを前提とする（**確認済み**: 対象サイトでアプリケーションパスワードの発行が可能であることをクライアントに確認済み）
- Googleスプレッドシート・Google Sheets APIの利用にはサービスアカウントを作成し、対象スプレッドシートへ編集権限を付与する
- LLM利用については、Claude Code CLI（`claude -p`）またはCowork＋ローカルrunnerサーバー方式により、既存のClaudeサブスクリプション（Pro/Max等）の範囲で実行する。従量課金のAnthropic APIキーは使用しない
- Claude Code CLI方式はClaudeサブスクリプションの使用枠（約5時間ごとにリセット）を消費するため、大量処理（店舗一括取得・400記事一括移行等）は夜間・休日などの実行を推奨する
- 記事の「紹介文」等、店舗情報表以外の部分については人間が引き続き執筆・編集するものとし、本システムの対象外とする
- 店舗情報の情報源は**HPBを第一優先**とする。HPBに記載がない項目のみ、登録済みの公式HP URLから補完する。公式HP URLが未登録、または公式HPにも該当情報がない場合、当該項目はnullのまま「要確認」として記録し、無理に埋めない
- 公式HP URLの収集・登録は、ツールAの入力（店舗URL一覧・店舗名一覧と合わせて任意項目として渡す）またはHPB店舗ページに公式HPへのリンクが掲載されている場合はそこから自動取得することを基本とする
- 初回の記事一括移行（ツールA'）は、以下の3段階を経てから全記事へ展開する
  1. **ローカルWPでの動作確認**: 本番記事の実データ（table/タブ/div/ショートコードの各パターンを含む数記事）をWordPress REST API（`?context=edit`で生の`post_content`を取得）またはエクスポート機能で取得し、開発者のローカルWP環境へ投入（REST API経由での投稿作成、またはClassic Editorのテキスト/コードビューでの貼り付け。Gutenbergのビジュアルエディタへの直接貼り付けは構造が変わる恐れがあるため行わない）。本番環境に一切影響を与えずに構造解析・置換ロジックを検証する
  2. **少数記事でのテスト実行**: 本番WPに対し、下書き保存モードで数記事のみ実際に反映し、担当者が目視で表示崩れ等を確認する
  3. **全記事への本番展開**: 1・2で問題がないことを確認してから、残り全記事へ展開する
- 差分検知後の記事反映は、変更履歴シートの`approved`列（チェックボックス）を担当者が手動でTRUEにすることを承認とする。スプレッドシートの編集権限を持つ担当者のみが承認操作を行える
- **【確認済み】** 対象WordPressサイトのエディタ形式は**標準のブロックエディター（Gutenberg）**であり、Elementor等のページビルダー系プラグインは使用していないことをクライアントに確認済み（SEO対策用の「AIOSEO」プラグインは導入されているが、記事構造には影響しない）。既存の構造解析ロジックでそのまま対応可能なため、追加の設計変更・工数は発生しない

## 11. リスク

| # | リスク | 影響度 | 対応方針 |
|---|---|---|---|
| 1 | HPBのHTML構造変更によりスクレイピングが失敗する | 高 | セレクタを設定ファイル化し局所修正可能にする。取得失敗時はエラーログを残し処理は継続（他店舗への影響を防ぐ） |
| 2 | HPB側のbot対策（アクセス制限・CAPTCHA等）により取得がブロックされる | 高 | アクセス間隔の調整、リトライ処理、User-Agent設定。頻度を抑えた定期実行とする |
| 3 | LLMによる情報抽出の誤判定（料金の読み違い等） | 中 | 抽出結果に確信度/フラグを付与し、低確信度の項目は自動反映せず人手確認に回す設計とする |
| 4 | 400記事の構造がツールA'の解析パターンに当てはまらない | 中 | 主要パターン（table/タブ/div/ショートコード）を網羅した上で、非対応パターンは「要確認」として自動更新をスキップしレポートする |
| 5 | WordPress記事の誤更新（紹介文まで書き換わる等） | 高 | 更新範囲を店舗情報部分に限定するホワイトリスト方式の置換ロジックとし、ドライラン確認を必須プロセスとする |
| 6 | Google Sheets APIのクォータ超過 | 低 | バッチ書き込み（batchUpdate）を用いてAPIコール数を削減する |
| 7 | APIキー等のシークレット漏洩 | 高 | プロジェクトフォルダ外のシークレットフォルダで一元管理し、リポジトリ・ログに含めない（[グローバルルール](../CLAUDE.md)準拠） |
| 8 | 公式HPの構造が店舗ごとに全くバラバラで自動抽出が困難 | 中 | 記事構造解析と同様にAIによる汎用抽出（キーワード近傍探索＋LLM判定）を用いる。抽出できない場合は無理に埋めずnullのまま「要確認」とする |
| 9 | HPBと公式HPで情報が食い違う（例: 料金が異なる） | 中 | HPB優先の原則を機械的に適用し、公式HPの値で上書きしない。食い違いが疑われる場合はother_info等に注記として記録し人手判断に委ねる |
| 10 | 変更履歴シートの`approved`列を誤ってTRUEにしてしまい、意図しない記事更新が発生する | 中 | 承認前に旧値/新値を必ず確認する運用とする。更新対象は店舗情報部分に限定されるため被害範囲は限定的で、WordPressのリビジョン機能で復元可能 |
| 11 | 閉店誤判定（一時的なサーバーエラー・Bot対策・HPB掲載終了を実際の閉店と誤検知する） | 高 | アクセスエラー（HTTP 404/403/タイムアウト等）およびHPBの「掲載期間が終了しました」は、それ単独では`closed`候補にしない（A-9-1/A-9-2参照）。`closed`候補とするのは公式HP・SNS等**店舗自身の発信**で閉店が明言されている場合に限定する。最終的な記事反映もapproved列での人手承認を必ず経る |

> リスク8・9は公式HPフォールバック追加に伴い、リスク10はSlack連携をスプレッドシート承認方式に変更したことに伴い、リスク11は閉店判定機能の追加に伴い今回新たに識別したもの。
