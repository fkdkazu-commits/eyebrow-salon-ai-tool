## 1. システム基本設計

### 1.1 全体アーキテクチャ

3ツールは共通基盤（scraper / llm / sheets / wordpress の各クライアント）を共有ライブラリとして参照する、モノレポ構成のCLIツール群とする。Slack連携はこれらのツールを呼び出す薄いフロントエンド（`apps/slack-bot`）として追加し、業種固有ロジックには関与しない。

```
                        ┌───────────────────────────┐
                        │   Slack ワークスペース      │
                        │  /tool-a /tool-a-prime     │
                        │  /tool-b コマンド           │
                        └──────────────┬─────────────┘
                                       │ Socket Mode (WebSocket, 常時待受)
                                       ▼
                        ┌───────────────────────────┐
                        │  apps/slack-bot (Bolt)      │
                        │  ・コマンド受付/権限チェック  │
                        │  ・各ツールの呼び出し        │

                        └──────────────┬─────────────┘
                                       │ 関数呼び出し / 子プロセス起動
                ┌────────────────────────────┐

                │  wordpress / logger / config│
```

### 1.3 ツール構成

| ツール | エントリーポイント | 主な依存パッケージ |
|---|---|---|
| ツールA | `apps/tool-a/src/main.ts` | playwright, googleapis, @anthropic-ai/sdk |
| ツールA' | `apps/tool-a-prime/src/main.ts` | cheerio, googleapis, @anthropic-ai/sdk, node-fetch |
| ツールB | `apps/tool-b/src/main.ts` | playwright, googleapis, @anthropic-ai/sdk |
| Slack連携 | `apps/slack-bot/src/main.ts` | @slack/bolt, execa（各ツールの子プロセス起動）|

各ツールは `packages/core` の共通クライアント（ScraperClient / LlmClient / SheetsClient / WordPressClient / Logger）を利用し、業種固有ロジック（HPB用セレクタ、抽出プロンプト）のみを各ツール側に持つ。これにより将来的に「業種B用ツール」を追加する際も、core層はそのまま流用できる。Slack連携もこの共通クライアント群には依存せず、各ツールをCLIサブプロセスとして起動し標準出力/終了コードから結果を受け取る疎結合な設計とする（ツール側の実装変更がSlack Bot側に影響しにくくするため）。

### 1.4 モジュール構成

```
packages/core
├── scraper/         # Playwright操作の共通化（ページ取得・リトライ・待機）
├── llm/              # LLM呼び出しの共通化（プロンプト実行・JSON検証・リトライ）
├── sheets/           # Google Sheets APIラッパー（読込・書込・batchUpdate・色付け）
├── wordpress/        # WordPress REST APIラッパー（記事取得・更新）
├── config/           # 設定・環境変数ローダー
└── logger/           # 構造化ログ出力（実行ログシート/ファイル）

apps/tool-a
├── crawler/          # HPB店舗ページ・クーポンページの巡回ロジック
├── extractor/        # LLMによる情報抽出（プロンプト・スキーマ）
├── fallback/         # 公式HP補完取得（不足項目検出→URL解決→取得→LLM抽出）
└── writer/           # 店舗マスタへの書き込みロジック（info_source_fields記録含む）

apps/tool-a-prime
├── article-fetcher/  # WordPress記事取得
├── structure-parser/ # table/タブ/div/ショートコード判定（HTML解析設計 参照）
├── matcher/          # 記事内店舗と店舗マスタの名寄せ（AI利用）
├── formatter/        # 統一フォーマットHTML生成
└── publisher/        # WordPress更新（ドライラン/本番）

apps/tool-b
├── crawler/          # ツールAのcrawlerを再利用
├── differ/           # 店舗マスタとの差分比較ロジック
└── history-writer/   # 変更履歴シートへの書き込み・セル色付け

apps/slack-bot
├── commands/         # /tool-a /tool-a-prime /tool-b の各コマンドハンドラ
├── auth/             # 実行可能チャンネル/ユーザーのホワイトリストチェック
├── runner/           # 対象CLIを子プロセスとして起動し標準出力を収集
└── notifier/         # 実行結果・差分検知サマリのSlackメッセージ整形・投稿
```

### 1.5 画面一覧

本システムはCLIツールであり画面（UI）は持たない。「画面」に相当するインターフェースは以下の通り。

| インターフェース | 種別 | 説明 |
|---|---|---|
| CLI実行結果（ターミナル出力） | CLI | 実行進捗・成否サマリを標準出力に表示 |
| 実行ログシート（Google Sheets） | シート | 過去の実行結果を一覧で確認 |
| 変更履歴シート（Google Sheets） | シート | 差分検知結果を確認する主要画面（担当者が日常的に見る） |
| 店舗マスタシート（Google Sheets） | シート | 店舗情報の正本を確認・手動補正する画面 |
| ツールA'実行レポート（Markdown/CSV出力） | ファイル | 記事更新の成功/失敗/要確認一覧を確認する画面 |
| Slackチャンネル | チャット | スラッシュコマンドでのツール起動と、実行結果・差分検知サマリの通知を受け取る主要な操作/確認インターフェース |

### 1.6 処理フロー

#### ツールA 処理フロー

```
START
 │
 ├─ 1. 入力読込（店舗URL一覧 or 店舗名一覧）
 ├─ 2. 店舗名のみの場合 → HPB検索でURL特定
 ├─ 3. 店舗ごとにループ:
 │     ├─ 3-1. Playwrightで店舗ページを開く
 │     ├─ 3-2. 基本情報をHTMLから抽出（DOM直接取得 or LLM補助）
 │     ├─ 3-3. クーポンページを開く
 │     ├─ 3-4. LLMでクーポン情報を判定・抽出（JSON形式で受領）
 │     ├─ 3-5. 抽出結果をバリデーション（必須項目・型チェック）
 │     ├─ 3-6. 必須項目にnullが残る場合 → 公式HP補完ステップへ（3-6-1〜）
 │     │     ├─ 3-6-1. 公式HP URLの確認（入力パラメータ or HPB店舗ページ内リンクから取得）
 │     │     ├─ 3-6-2. URLが無い/取得失敗 → 補完せずnullのまま「要確認」フラグを立てる
 │     │     └─ 3-6-3. URLがある場合 → Playwrightで公式HPを取得しLLMでnull項目のみ抽出
 │     │           （HPBに値がある項目は上書きしない。抽出できた項目のinfo_sourceを`official_site`に設定）
 │     ├─ 3-7. 失敗時: リトライ（最大3回）→ 失敗ログ記録しスキップ
 │     └─ 3-8. 成功時: 内部データ配列に追加（各項目のinfo_source: `hpb`/`official_site`/`null`を付与）
 ├─ 4. 全店舗分をGoogle Sheets batchUpdateで一括書き込み
 ├─ 5. 実行ログシートへサマリ（公式HP補完件数を含む）記録
 └─ END
```

> **情報源の優先順位**: HPBを第一情報源とし、HPBに記載のある項目は公式HPの値で上書きしない。公式HPからの補完はHPBに値が存在しない項目に限定する。

#### ツールA' 処理フロー

```
START
 │
 ├─ 1. WordPress REST APIで対象記事一覧を取得
 ├─ 2. 記事ごとにループ:
 │     ├─ 2-1. 記事本文HTML取得
 │     ├─ 2-2. 構造解析（HTML解析設計のロジックで店舗情報ブロックを検出）
 │     ├─ 2-3. 検出失敗 → 「要確認」リストへ追加しスキップ
 │     ├─ 2-4. ブロックごとに店舗名を抽出しAIで店舗マスタと名寄せ
 │     ├─ 2-5. 名寄せ確信度が閾値未満 → 「要確認」リストへ追加しスキップ
 │     ├─ 2-6. 店舗マスタの最新情報で統一フォーマットHTMLを生成
 │     ├─ 2-7. 元HTML内の該当ブロックのみを新HTMLに置換（紹介文等は非破壊）
 │     ├─ 2-8. ドライランモード時: 差分をレポートに出力のみ（更新しない）
 │     └─ 2-9. 本番モード時: WordPress REST APIで記事を更新（下書き保存 or 公開）
 ├─ 3. 実行結果レポート（成功/失敗/要確認）を出力
 └─ END
```

#### ツールB 処理フロー

```
START
 │
 ├─ 1. 店舗マスタシートから全店舗（店舗URL含む）を取得
 ├─ 2. 店舗ごとにループ:
 │     ├─ 2-1. Playwrightで店舗ページ/クーポンページを再取得
 │     ├─ 2-2. ツールAと同じ抽出ロジックで最新情報を取得
 │     ├─ 2-3. マスタの現在値と項目単位で比較
 │     ├─ 2-4. 差分あり → 該当セルを黄色に色付け + 変更履歴シートへ追記
 │     ├─ 2-5. 取得失敗 → リトライ後、失敗ログを記録しスキップ（マスタは変更しない）
 │     └─ 2-6. 差分なし → 何もしない
 ├─ 3. 実行ログシートへサマリ（巡回数/差分数/失敗数）記録
 └─ END
```

#### Slack連携 処理フロー

```
START
 │
 ├─ 1. Slackスラッシュコマンド受信（/tool-a, /tool-a-prime, /tool-b）
 ├─ 2. 実行可能チャンネル/ユーザーのホワイトリストチェック
 │     └─ NG → 「権限がありません」を即時応答し終了
 ├─ 3. Slackへ即時応答（「実行を開始しました」。3秒以内制約への対応）
 ├─ 4. 対象CLIを子プロセスとして非同期起動（引数はコマンドパラメータから組立）
 ├─ 5. 子プロセスの標準出力/終了コードを収集
 ├─ 6. 終了コードに応じて結果を整形
 │     ├─ 正常終了 → 成功/失敗/要確認件数のサマリを作成
 │     └─ 異常終了 → エラー概要（ログファイルパス付き）を作成
 └─ 7. Slack Web API（chat.postMessage）でコマンド実行元チャンネルへ結果を投稿
 END
```

### 1.8 エラー処理方針

| レイヤ | エラー種別 | 方針 |
|---|---|---|
| スクレイピング（Playwright） | ページ取得失敗・タイムアウト | 指数バックオフで最大3回リトライ → 失敗時はログ記録しその店舗のみスキップ（全体処理は継続） |
| スクレイピング | HPB側のレイアウト変更でセレクタが取得できない | 該当項目をnull扱いにし「要確認」フラグを立てる。処理全体は止めない |
| LLM呼び出し | API呼び出しエラー（レート制限・タイムアウト等） | 指数バックオフで最大3回リトライ。恒久失敗時はログ記録しスキップ |
| LLM呼び出し | 出力がJSONスキーマに一致しない | 1回だけ「JSON形式を修正して再出力して」と再依頼 → それでも失敗なら手動確認行き |
| Google Sheets API | クォータ超過・書込エラー | 指数バックオフでリトライ。batchUpdateにより1回のAPIコールに複数更新をまとめクォータ消費を抑制 |
| WordPress REST API | 認証エラー | 即座に処理を中断し、認証情報の確認を促すエラーメッセージを表示（誤った記事更新を防止） |
| WordPress REST API | 更新競合（記事が同時編集中等） | 更新前にETag/更新日時を確認し、競合時はスキップして「要確認」に記録 |
| 共通 | 想定外の例外 | 個別店舗・個別記事単位でtry-catchし、1件の失敗が全体を止めないようにする。最終的にサマリで失敗件数を報告 |
| Slack連携 | 許可外チャンネル/ユーザーからのコマンド実行 | 実行せず「権限がありません」と即時応答。ログにも記録し悪用検知に備える |
| Slack連携 | Socket Mode接続断（ネットワーク瞬断等） | Bolt SDKの自動再接続に任せる。再接続失敗が続く場合は監視ログにエラーを記録し、CLI直接実行へフォールバック可能なことを運用手順に明記 |
| Slack連携 | 対象CLIプロセスの異常終了・タイムアウト | 終了コード/タイムアウトを検知し、エラー概要とログファイルパスをSlackへ通知。Slack Bot自体は落とさない |
| Slack連携 | Slack Web API呼び出し失敗（通知送信エラー） | リトライ後、失敗時はログファイルに結果を残し「Slack通知に失敗しました。ログを確認してください」旨を次回コマンド応答時等に補足できるようにする |

## 2. データ設計

Googleスプレッドシート1ファイルに以下4シートを持つ構成とする。

### 2.1 シート一覧

| シート名 | 用途 |
|---|---|
| 店舗マスタ | HPBを正とした店舗情報の正本データ |
| 変更履歴 | ツールBが検知した差分の履歴 |
| ログ | ツールA/A'/Bの実行ログ |

## 4. AI設計

### 4.1 LLM利用箇所と使い分け

| 利用箇所 | ツール | 目的 | 入力 | 出力 |
|---|---|---|---|---|
| クーポン情報抽出 | A | クーポンページの自由なレイアウトから料金・人気メニュー等を判定 | クーポンページのテキスト/HTML抜粋 | JSON（料金・学割等） |
| 公式HP補完抽出 | A | HPBに記載がない項目のみを対象に、店舗ごとに構造の異なる公式HPから該当情報を判定・抽出 | 公式HPのテキスト/HTML抜粋＋不足項目リスト | JSON（不足項目の値・confidence） |
| 記事構造解析 | A' | table/タブ/div等の店舗情報ブロックを特定 | 記事本文HTML | JSON（ブロック位置・種別） |
| 店舗名寄せ | A' | 記事内の店舗表記とマスタ店舗を同定 | 記事内店舗名候補＋マスタ店舗リスト | JSON（store_id・confidence） |
| 差分の意味判定（任意） | B | 表記揺れ（例:「10:00-20:00」→「10:00〜20:00」）を実質同一と判定し誤検知を抑制 | 旧値・新値のペア | JSON（is_meaningful_change: boolean） |

### 4.2 プロンプト設計

各プロンプトは「役割定義 → 制約 → 出力形式（JSON Schema）→ 入力データ」の順で構成する共通テンプレートを`packages/core/llm/prompts/`に配置する。

**例: クーポン情報抽出プロンプト（ツールA）**

```
あなたは美容サロンのクーポンページから料金情報を正確に抽出するアシスタントです。

# 制約
- 記載されている情報のみを使用し、推測で値を作らないこと
- 該当情報が見つからない項目は null にすること
- 金額は数値（円、税込表記があれば税込優先）のみを抽出し、単位や記号は含めないこと
- 「初回」「はじめて」「新規」等の表記があるクーポンを first_visit_price として扱うこと
- 「メンズ」「男性」の表記があるクーポンを mens_price として扱うこと

# 出力形式（JSON以外は出力しないこと）
{
  "first_visit_price": number | null,
  "regular_price": number | null,
  "mens_price": number | null,
  "student_discount": string | null,
  "popular_menu": string | null,
  "confidence": number  // 0.0〜1.0
}

# 入力（クーポンページ抜粋）
<<<INPUT_HTML_TEXT>>>
```

**例: 公式HP補完抽出プロンプト（ツールA）**

```
あなたは美容サロンの公式ホームページから、指定された不足項目のみを正確に抽出するアシスタントです。
このサイトはホットペッパービューティーには掲載されていない情報を補完する目的で参照しています。

# 制約
- 指定された不足項目リストに含まれる項目のみを抽出すること（それ以外の項目は出力しないこと）
- 記載されている情報のみを使用し、推測で値を作らないこと
- 該当情報が見つからない項目は null にすること
- ページ内に複数店舗（他支店）の情報が混在する場合、対象店舗（下記store_nameで指定）の情報のみを抽出すること

# 出力形式（JSON以外は出力しないこと。missing_fieldsに無い項目のキーは含めないこと）
{
  "<不足項目名>": "<抽出値>" | null,
  ...
  "confidence": number  // 0.0〜1.0
}

# 入力
対象店舗名: <<<STORE_NAME>>>
不足項目リスト（missing_fields）: <<<MISSING_FIELDS_JSON>>>
公式HPページ抜粋: <<<INPUT_HTML_TEXT>>>
```

**例: 店舗名寄せプロンプト（ツールA'）**

```
あなたは記事内に記載された店舗表記と、店舗マスタの正式な店舗情報を照合するアシスタントです。

# 制約
- 店名の表記揺れ（支店名の省略・全角半角・スペース等）を考慮すること
- 住所・電話番号が一致する場合は強い根拠として扱うこと
- 確信が持てない場合は無理に一致させず confidence を低くすること

# 出力形式（JSON以外は出力しないこと）
{
  "matched_store_id": string | null,
  "confidence": number, // 0.0〜1.0
  "reason": string
}

# 入力
記事内店舗表記: <<<ARTICLE_STORE_TEXT>>>
店舗マスタ候補リスト: <<<CANDIDATE_STORES_JSON>>>
```

### 4.4 リトライ方法

1. LLM呼び出し失敗（レート制限・タイムアウト・5xx）: 指数バックオフ（1秒 → 2秒 → 4秒）で最大3回リトライ
2. 出力がJSONとしてパース不能、またはスキーマ不一致: エラー内容を添えて「先ほどの出力はJSON形式として不正でした。スキーマに厳密に従いJSONのみを再出力してください」と1回だけ再依頼
3. 上記すべて失敗: 当該項目を`null`かつ`confidence: 0`として扱い、「要確認」リストに記録して処理は継続（全体を止めない）

### 4.5 トークン節約方法

- クーポンページ抽出時は、ページ全体のHTMLではなく、事前にcheerio等で不要なタグ（script/style/nav/footer等）を除去したテキスト抜粋のみをLLMに渡す
- 記事構造解析は、本文HTML全体ではなく店舗情報が含まれる可能性の高いブロック（見出し「店舗情報」「アクセス」等の近傍）に絞り込んでから渡す
- プロンプトのシステム部分（役割定義・制約・出力形式）は固定文言とし、プロンプトキャッシュ（Claude APIのprompt caching）を活用してコストを削減する
- 店舗名寄せでは、店舗マスタ全件ではなく、都道府県・エリア等で事前フィルタした候補リストのみをLLMに渡す
- モデル選定は[グローバルルール](../CLAUDE.md#ai機能実装時のモデル選定ルール)に従い、まずHaikuで品質検証を行った上で、品質要件を満たせばHaikuを採用しコストを抑制する（未検証の状態で本番投入しない）

## 5. HTML解析設計

400記事に存在しうる店舗情報の構造パターンを以下に整理する。ツールA'の`structure-parser`はこれらのパターンを優先度順に判定し、いずれにも一致しない場合はAIによる汎用解析にフォールバックする。

### 5.1 パターン一覧

| パターン | 特徴 | 検出方法 |
|---|---|---|
| ① table形式 | `<table>`内に「店舗名/住所/営業時間」等の行・列がある | `<table>`要素を走査し、ヘッダ行またはth要素に店舗情報系キーワード（住所・営業時間・定休日等）を含むものを抽出対象と判定 |
| ② タブ形式 | 店舗ごとにタブ切り替え（例: WP Tabs系プラグイン、独自div+JS） | タブUIに使われる典型クラス名（`tab`, `tabs`, `wp-block-tabs`等）とタブ内に店舗情報キーワードが含まれるかで判定 |
| ③ div形式（カード型） | 店舗ごとに`<div class="shop-card">`等で情報が箇条書き | 繰り返し構造（同一クラスを持つdivが複数存在）かつ内部に店舗情報キーワードを含むブロックを抽出対象と判定 |
| ④ ショートコード形式 | `[shop_info id="123"]`等、WordPressショートコードで店舗情報を差し込み | 正規表現でショートコードパターンを検出し、IDパラメータから対象店舗を特定（この場合はショートコード自体は書き換えず、ショートコードが参照するカスタムフィールド/テーブル側を更新する方式も選択肢とする） |
| ⑤ 非対応・不明パターン | 上記いずれにも一致しない | 「要確認」としてスキップし、レポートに記事URLと該当箇所候補を出力。人手でフォーマットAへの移行 or パターン追加を判断 |

### 5.2 検出アルゴリズム（概要）

```
1. 記事本文HTMLをcheerioでパース
2. 見出し（h2/h3/strong等）に「店舗情報」「アクセス」「基本情報」等の
   キーワードが含まれる箇所を候補セクションとしてマーキング
3. 候補セクション内の直下・近傍の構造を判定
   ├─ <table> が存在 → パターン①
   ├─ タブ系クラス名が存在 → パターン②
   ├─ 同一クラスの繰り返しdivが存在 → パターン③
   ├─ ショートコード文字列が存在 → パターン④
   └─ 上記非該当 → AIに構造判定を依頼（該当箇所のHTML断片を渡し
      「これは店舗情報ブロックか、どのパターンに近いか」をJSONで判定させる）
4. なお特定不可 → パターン⑤（要確認）としてスキップ
```

### 5.3 置換ルール（共通）

- 検出したブロックの**開始タグ〜終了タグ全体**をDOM単位で新フォーマットHTMLに置換する（テキストの部分書き換えは行わない。誤爆防止のため）
- ブロックの前後にある紹介文・見出しテキスト自体は変更しない（見出し文言はブロックの外側とみなす）
- 置換後HTMLには目印として`<!-- shop-info-block:store_id=H000123456 -->`のHTMLコメントを付与し、次回以降のツールA'実行時に該当ブロックを高速・確実に再検出できるようにする（統一フォーマット移行後の2回目以降はこのコメントを最優先で検出）

## 6. 差分検知設計

### 6.1 比較ロジック

1. ツールBが取得した最新情報（フィールドごとの値）と、店舗マスタの現在値を**フィールド単位**で文字列比較する
2. 単純な文字列不一致だけでなく、表記揺れ（全角半角、ハイフンの種類、スペース等）を吸収する正規化処理を比較前に適用する
   - 例: `10:00-20:00` と `10:00〜20:00` は正規化後に同一とみなし差分なしとする
3. 正規化後も不一致の場合のみ「実質的な差分」として扱う
4. 任意オプションとして、正規化だけでは判断が難しい微妙なケース（4.1のAI差分意味判定）をAIに委ねる設定を用意する（既定OFF、設定シートで有効化可能）
5. **情報源の優先順位はツールAと同様に維持する**: `info_source_fields`にフィールド名が記録されている項目（=公式HP由来）は、再取得時もHPBではなく公式HPを情報源として再取得・比較する。`info_source_fields`に記録がない項目（=HPB由来）はHPBを再取得・比較する

### 6.2 比較対象フィールド

店舗マスタの`business_hours` `closed_days` `access` `phone_number` `payment_methods` `parking` `first_visit_price` `regular_price` `mens_price` `student_discount` `popular_menu` `other_info` を比較対象とする。

### 6.3 比較対象外フィールド

`store_id`（固定キー）、`hpb_url`（固定）、`review_count` `rating`（頻繁に微変動するため差分検知の対象外とし、値の更新のみ行い黄色ハイライト・履歴記録はしない）、`last_checked_at` `last_updated_at` `status`（システム内部管理用のメタ情報）。

> 口コミ件数・評価は日々変動するため、これを差分として毎回ハイライト・履歴記録すると変更履歴シートがノイズだらけになる。値自体はマスタに反映しつつ、「注目すべき差分」からは除外する。

### 6.4 色付けルール

| 条件 | 色 |
|---|---|
| 比較対象フィールドに実質的な差分あり | 黄色（`#FFFF00`）背景をセルに設定 |
| 差分なし | 色付けなし（既存の書式のまま） |
| 前回ハイライトされたセルで、担当者が確認済みとして手動で色をクリアした場合 | ツール側は再度上書きしない（担当者操作を尊重し、次回巡回で新たな差分があった場合のみ再度色付け） |

### 6.5 履歴作成ルール

- 実質的な差分を検知したフィールドごとに、変更履歴シートへ**1行追記**する（複数フィールドが同時に変わった場合は複数行）
- 履歴は追記のみとし、既存行の削除・上書きは行わない（監査目的のため）
- `reflected_to_articles`列は既定`FALSE`とし、ツールA'によって当該店舗を含む記事の再フォーマットが行われた際に`TRUE`へ更新する（ツールA'とツールBを紐付ける唯一の連携ポイント）

### 9.4 Git管理

`.gitignore` に以下を含める（[グローバルルール](../CLAUDE.md#セキュリティルール)準拠）:

```
.env
node_modules/
logs/
reports/
*.log
```

- `main`ブランチ: 常にデプロイ可能な状態を維持
- `feature/*`: 機能単位の開発ブランチ（例: `feature/tool-a-crawler`）
- `docs/*`: ドキュメント整備用ブランチ（本プロジェクトの要件定義・設計書はこの形式で作成・公開）
- コミット前に、APIキーのハードコードがないか・`.env`が誤ってステージされていないかを必ず確認する
