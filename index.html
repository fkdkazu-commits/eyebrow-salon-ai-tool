01背景・目的

なぜ今、店舗情報の自動化が必要なのか

背景

現在400記事以上を保有し、各記事に複数店舗の情報（住所・営業時間・料金など）が掲載されています。本来はホットペッパービューティー（HPB）の情報を正とすべきですが、マスタとなるデータが存在せず、記事ごとに手作業で入力されているため、情報の鮮度・正確性が担保できない状態です。

目的

  1. HPBを正とした店舗マスタをGoogleスプレッドシート上に構築し、一元管理する
  2. 400記事以上に散在する店舗情報表を、AIで統一フォーマットへ一括置換する
  3. HPB側の情報変更を定期的に自動検知し、マスタ更新と変更履歴の記録を自動化する
  4. 情報の正確性・鮮度を維持しながら運用工数を大幅に削減する

02課題

課題影響
店舗マスタが存在しない情報の正本がなく、記事間で表記揺れ・食い違いが発生
記事構造が不統一（table / タブ / div等）一括置換・機械的更新ができない
HPB更新を検知する仕組みがない古い営業時間・料金がユーザーに提示され続ける
400記事×複数店舗の手動確認は工数過多更新が後回しになり鮮度がさらに悪化
変更履歴が残らないいつ何が変わったか追跡できず、誤更新の検証もできない

03業務フロー（As-Is → To-Be）

As-Is（現状）

HPBの店舗情報が変更される
   → 誰も気づかない
   → 気づいたら手動で記事を検索
   → 目視でHPBと見比べ、手動でHTML編集
   → 同じ店舗が載った記事の数だけ繰り返し
問題: 「変更に気づく」起点が存在しないため、実際にはほとんど更新されません。

To-Be（自動化後）

■ 初期構築（ツールA）
店舗URL/店舗名一覧 → AIがHPBを巡回・抽出 → 店舗マスタに自動登録

■ 記事統一フォーマット化（ツールA'）
既存400記事を巡回 → AIが構造(table/タブ/div等)を解析
→ 店舗マスタの最新情報で統一フォーマットへ部分置換
→ WordPress REST APIで更新（紹介文は変更しない）

■ 定常運用（ツールB・定期実行）
HPBを巡回 → 店舗マスタと差分検知
→ 差分セルを黄色にハイライト
→ 変更履歴シートに 日時/店舗/項目/旧値/新値 を記録
→ 担当者が確認し、必要に応じてツールA'で記事へ反映

04システム概要

Claude Code（Node.js / TypeScript）で開発する、独立した3つのCLIツールで構成します。共通のスクレイピング基盤（Playwright）・LLM抽出基盤・Google Sheets連携基盤を共有ライブラリとして再利用します。

ツールA — 店舗マスタ作成

  • 店舗URL/店舗名一覧を入力
  • Playwrightで店舗ページ・クーポンページを取得
  • LLMで料金・人気メニュー等を判定抽出
  • Google Sheetsに1店舗1行で登録

ツールA' — 記事フォーマット統一

  • WordPress REST APIで対象記事を取得
  • AIが構造(table/タブ/div等)を解析
  • 店舗マスタ最新情報で統一フォーマット生成
  • 店舗情報部分のみ部分置換して更新

ツールB — 差分検知

  • 店舗マスタ登録店舗を定期巡回
  • 最新情報とマスタを項目単位で比較
  • 差分セルを黄色にハイライト
  • 変更履歴シートに記録

共通基盤

  • Scraperクライアント（Playwright）
  • LLMクライアント（Claude API）
  • Sheetsクライアント（Google Sheets API）
  • WordPressクライアント / Logger

06非機能要件・制約事項

項目内容
実行環境Windows / Node.js(LTS) / TypeScript / CLI実行
対象データ量記事約400件、店舗数は数百規模
スクレイピング耐性HPBのレイアウト変更・アクセス制限に対しリトライ・待機・UA設定で耐性確保
データ整合性Google Sheetsを唯一の正(Single Source of Truth)とし記事側は一方向反映
セキュリティAPIキー等はプロジェクトフォルダ外の専用シークレットフォルダで管理
拡張性業種固有ロジックを設定・プロンプトに分離し、他業種メディアへ横展開可能な設計
制約: Playwright / Google Sheets API / WordPress REST API / Git の利用が必須。HPBの利用規約・robots.txtの範囲内でのアクセスに限定します。

07主なリスクと対応方針

リスク影響度対応方針
HPBのHTML構造変更でスクレイピング失敗高セレクタを設定ファイル化し局所修正可能に。失敗は個別スキップし全体は継続
HPBのbot対策によるブロック高アクセス間隔調整・リトライ・頻度を抑えた定期実行
LLMの誤判定(料金の読み違い等)中確信度(confidence)を付与し、低確信度は自動反映せず人手確認へ
WordPress記事の誤更新高店舗情報部分に限定した置換ロジック＋ドライラン確認を必須化
APIキー等の漏洩高専用シークレットフォルダで一元管理し、リポジトリ・ログに含めない

09データ設計（Google Sheets）

1スプレッドシートに4シート構成。店舗マスタを正本とし、変更履歴は追記のみで監査性を担保します。

シート用途主なカラム例
店舗マスタ店舗情報の正本(1店舗1行)store_id / store_name / address / business_hours / regular_price / status など全21列
変更履歴ツールBが検知した差分の履歴(追記のみ)detected_at / store_id / field / old_value / new_value / reflected_to_articles
ログ各ツールの実行ログrun_at / tool / success_count / fail_count / diff_count
設定各ツールの動作パラメータSCRAPE_INTERVAL_MS / RETRY_MAX / MATCH_CONFIDENCE_THRESHOLD

10API設計

API用途認証方式
Google Sheets API店舗マスタ・履歴・ログの読み書き、セル色付けサービスアカウント(OAuth2)
WordPress REST API記事一覧・本文取得、店舗情報部分の更新アプリケーションパスワード(Basic認証)
PlaywrightHPB店舗ページ・クーポンページの取得－（公開ページの取得）
Claude API店舗情報抽出・記事構造解析・店舗名寄せ判定APIキー(Bearer認証)
全ての認証情報はプロジェクトフォルダ外の専用シークレットフォルダで管理し、リポジトリには .env.example（キー名のみ）を含めます。

11AI設計

LLM利用箇所

利用箇所目的出力
クーポン情報抽出(ツールA)料金・人気メニュー等の判定JSON(price/discount/confidence)
記事構造解析(ツールA')店舗情報ブロックの特定JSON(ブロック位置・種別)
店舗名寄せ(ツールA')記事内表記とマスタ店舗の同定JSON(store_id・confidence)

プロンプト設計・出力例

{
  "first_visit_price": 2980,
  "regular_price": 4980,
  "mens_price": 3980,
  "student_discount": "学生証提示で10%OFF",
  "popular_menu": "眉毛WAX＋トリートメント",
  "confidence": 0.93
}

リトライ・トークン節約

  • API失敗時は指数バックオフで最大3回リトライ、JSON不正時は1回だけ再依頼
  • 不要なHTMLタグを除去したテキストのみをLLMに渡す
  • プロンプトのシステム部分を固定化しprompt cachingを活用
  • 品質検証済みであればHaikuを優先しコストを抑制（グローバルルール準拠）

12HTML解析設計

400記事に存在しうる店舗情報の構造パターンを4種類+非対応パターンに整理し、優先度順に判定します。

パターン特徴
① table形式<table>内に店舗情報の行・列がある
② タブ形式店舗ごとにタブ切り替え
③ div形式(カード型)店舗ごとにdivカードで箇条書き
④ ショートコード形式[shop_info id="123"]形式
⑤ 非対応・不明「要確認」としてスキップし人手判断へ
検出したブロックは開始タグ〜終了タグ全体をDOM単位で置換し、紹介文などブロック外のテキストは変更しません。置換後は目印コメントを付与し次回実行を高速化します。

13差分検知設計

  • フィールド単位で比較し、全角半角等の表記揺れは正規化して吸収（誤検知防止）
  • 比較対象外: store_id / hpb_url / review_count / rating（頻繁に微変動するためノイズ防止）
  • 実質的な差分があったセルのみ黄色(#FFFF00)にハイライト
  • 差分は変更履歴シートへ追記のみ（削除・上書きしない＝監査性の担保）

14テスト方針

各ツールについて正常系・異常系・境界値・API異常・取得失敗・タイムアウトを網羅したテストケースを設計書に整理しています（合計31ケース）。

  • ツールA: 11ケース（新規登録・重複更新・クーポンなし店舗・URL不正・API異常 等）
  • ツールA': 11ケース（table/タブ/div各パターン・ドライラン・要確認判定・更新競合 等）
  • ツールB: 9ケース（差分検知・表記揺れ誤検知防止・閉店検知・API異常 等）

詳細は 設計書.md 7章 を参照してください。
