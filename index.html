眉毛サロン比較メディア
店舗情報AI自動更新システム

400記事以上に散在する店舗情報を、ホットペッパービューティーを第一情報源としてAIで自動収集・統一フォーマット化・差分検知するための3ツール＋Slack連携の要件定義・設計資料です。
1

背景・目的

なぜ今、店舗情報の自動化が必要なのか

背景

現在400記事以上を保有し、各記事に複数店舗の情報（住所・営業時間・料金など）が掲載されています。本来はホットペッパービューティー（以下HPB）の情報を正とすべきですが、マスタとなるデータが存在せず、記事ごとに手作業で入力されているため、情報の鮮度・正確性が担保できない状態です。

目的

  1. HPBを正とし、必要に応じて各店舗の公式HPで補完した店舗マスタをGoogleスプレッドシート上に構築し、一元管理する
  2. 400記事以上に散在する店舗情報表を、AIで統一フォーマットへ一括置換する
  3. HPB側の情報変更を定期的に自動検知し、マスタ更新と変更履歴の記録を自動化する
  4. 情報の正確性・鮮度を維持しながら運用工数を大幅に削減する
2

課題

As-Isの問題点
課題影響
店舗マスタが存在しない情報の正本がなく、記事間で表記揺れ・食い違いが発生
記事構造が不統一（table / タブ / div等）一括置換・機械的更新ができない
HPB更新を検知する仕組みがない古い営業時間・料金がユーザーに提示され続ける
400記事×複数店舗の手動確認は工数過多更新が後回しになり鮮度がさらに悪化
変更履歴が残らないいつ何が変わったか追跡できず、誤更新の検証もできない
3

業務フロー

As-Is（現状） → To-Be（自動化後）

As-Is（現状）

HPBの店舗情報が変更される → 誰も気づかない → 気づいたら手動で記事を検索 → 目視でHPBと見比べ、手動でHTML編集 → 同じ店舗が載った記事の数だけ繰り返し
⛔
問題: 「変更に気づく」起点が存在しないため、実際にはほとんど更新されません。

To-Be（自動化後）

■ 初期構築（ツールA） 店舗URL/店舗名一覧 → HPBを第一情報源としてAIが抽出 → 不足項目のみ公式HPで補完 → 店舗マスタに自動登録 ■ 記事統一フォーマット化（ツールA'） 既存400記事を巡回 → AIが構造(table/タブ/div等)を解析 → 店舗マスタの最新情報で統一フォーマットへ部分置換 → WordPress REST APIで更新（紹介文は変更しない） ■ 定常運用（ツールB・定期実行 + Slackからの随時実行） HPB(または公式HP)を巡回 → 店舗マスタと差分検知 → 差分セルを黄色にハイライト → 変更履歴シートに 日時/店舗/項目/旧値/新値 を記録 → Slackへ差分検知サマリを通知 → 担当者が確認し、必要に応じてツールA'で記事へ反映

システム概要

3つのCLIツール + Slack連携

Claude Code（Node.js / TypeScript）で開発する、独立した3つのCLIツールとSlack連携で構成します。共通のスクレイピング基盤（Playwright）・LLM抽出基盤・Google Sheets連携基盤を共有ライブラリとして再利用します。

ツール役割起動方法
ツールAHPBを第一情報源として店舗情報を取得し、不足項目は公式HPで補完して店舗マスタを新規構築するCLI / Slack
ツールA'既存記事内の店舗情報表を、店舗マスタを参照して統一フォーマットへ置換するCLI / Slack
ツールBHPB（または公式HP）を定期巡回し、店舗マスタとの差分を検知・記録するスケジューラ / Slack
Slack連携スラッシュコマンドで各ツールを起動し、実行結果・差分検知内容を通知する常時待受（Socket Mode）

情報源の優先順位

ホットペッパービューティー優先、不足時は公式HPで補完
ℹ️
基本方針：店舗情報（住所・料金・その他）はホットペッパービューティー（HPB）を第一情報源とします。HPBに記載がない項目のみ、各店舗の公式ホームページを補完的な情報源として利用します。HPBに値がある項目を公式HPの値で上書きすることはありません。

優先順位の判定フロー

対象項目（例：営業時間）を取得 │ ├─ HPBに記載あり ───────────→ HPBの値を採用（info_source = hpb） │ └─ HPBに記載なし │ ├─ 公式HP URLが登録されている │ │ │ ├─ 公式HPに記載あり → 公式HPの値を採用（info_source = official_site） │ └─ 公式HPに記載なし → null のまま「要確認」に記録 │ └─ 公式HP URLが未登録 → null のまま「要確認」に記録（無理に埋めない）

ルール

ルール内容
優先順位HPB > 公式HP。HPBの値は公式HPの値で上書きしない
公式HP URLの取得入力時に任意項目として渡す、またはHPB店舗ページ内のリンクから自動取得する
情報源の記録店舗マスタのinfo_source_fields列に、公式HPで補完した項目のみを記録（記載のない項目はすべてHPB由来）
両方に情報がない場合該当項目はnullのまま「要確認」として記録し、無理に埋めない
ツールBの再取得公式HP由来の項目（info_source_fieldsに記録済み）は、差分検知時も公式HPを再取得して比較する
⚠️
公式HPは店舗ごとに構造が大きく異なるため、記事構造解析と同様にAIによる汎用抽出（キーワード近傍探索＋LLM判定）で対応します。抽出できない場合は無理に値を埋めず「要確認」として人手判断に委ねます。
7

Slack連携（起動・通知）

Slack上のスラッシュコマンドで各ツールを起動し、結果を通知する

CLI手動実行に加えて、Slackのスラッシュコマンドから各ツールを起動できるようにします。担当者は普段使っているSlack上で操作でき、実行結果もそのままSlackで確認できます。

機能概要
スラッシュコマンド受付/tool-a /tool-a-prime /tool-b でツールを起動
即時応答Slackの3秒応答制約に合わせ「実行を開始しました」と即座に応答し、本処理は非同期実行
実行権限チェック許可されたチャンネル/ユーザーのみ実行可能（誤操作・悪用防止）
実行結果通知成功/失敗/要確認件数のサマリをSlackチャンネルへ投稿
差分検知通知ツールBが差分を検知した場合、変更店舗数・主な変更内容を要約して通知

起動フロー

Slackユーザー → /tool-b コマンド送信 → apps/slack-bot が権限チェック → 即時応答「実行を開始しました」 → ツールBを子プロセスとして非同期起動 → 完了後、結果サマリ(成功/失敗/差分件数)を chat.postMessage でSlackへ投稿
✅
可用性: Slack連携はSocket Mode（常時接続・公開エンドポイント不要）で動作しますが、Bot停止中もCLI直接実行は可能な設計とし、Slackを唯一の起動経路にはしません。

主なリスクと対応方針

全11件を識別（うちSlack連携2件、公式HPフォールバック2件）
リスク影響度対応方針
HPBのHTML構造変更でスクレイピング失敗高セレクタを設定ファイル化し局所修正可能に。失敗は個別スキップし全体は継続
HPBのbot対策によるブロック高アクセス間隔調整・リトライ・頻度を抑えた定期実行
LLMの誤判定(料金の読み違い等)中確信度(confidence)を付与し、低確信度は自動反映せず人手確認へ
WordPress記事の誤更新高店舗情報部分に限定した置換ロジック＋ドライラン確認を必須化
APIキー等の漏洩高専用シークレットフォルダで一元管理し、リポジトリ・ログに含めない
Slack経由の意図しない本番実行高実行可能チャンネル/ユーザーをホワイトリスト化。本番反映系コマンドは確認ステップを挟む
Slack Bot Token等の漏洩高シークレットフォルダで一元管理し、権限スコープを必要最小限に絞る
公式HPの構造が店舗ごとにバラバラで抽出困難中AIによる汎用抽出（キーワード近傍探索＋LLM判定）を使用。抽出不可時はnullのまま「要確認」
HPBと公式HPで情報が食い違う中HPB優先の原則を機械的に適用し、公式HPの値で上書きしない

データ設計（Google Sheets）

4シート構成

1スプレッドシートに4シート構成。店舗マスタを正本とし、変更履歴は追記のみで監査性を担保します。

シート用途主なカラム例
店舗マスタ店舗情報の正本(1店舗1行)store_id / hpb_url / official_site_url / business_hours / info_source_fields など全23列
変更履歴ツールBが検知した差分の履歴(追記のみ)detected_at / store_id / field / old_value / new_value / reflected_to_articles
ログ各ツールの実行ログrun_at / tool / success_count / fail_count / diff_count / triggered_by
設定各ツールの動作パラメータSCRAPE_INTERVAL_MS / RETRY_MAX / MATCH_CONFIDENCE_THRESHOLD / SLACK_ALLOWED_CHANNEL_IDS
📝
official_site_url＝店舗の公式HP（任意項目）。info_source_fields＝HPBに情報がなく公式HPで補完した項目のみを記録（記載のない項目はすべてHPB由来）。

リトライ・トークン節約

  • API失敗時は指数バックオフで最大3回リトライ、JSON不正時は1回だけ再依頼
  • 不要なHTMLタグを除去したテキストのみをLLMに渡す
  • 公式HP補完抽出は不足項目のみを対象にし、既にHPBで取得済みの項目は再度LLMに渡さない
  • プロンプトのシステム部分を固定化しprompt cachingを活用
  • 品質検証済みであればHaikuを優先しコストを抑制（グローバルルール準拠）

HTML解析設計

記事内の店舗情報ブロック特定パターン

400記事に存在しうる店舗情報の構造パターンを4種類+非対応パターンに整理し、優先度順に判定します。

パターン特徴
① table形式<table>内に店舗情報の行・列がある
② タブ形式店舗ごとにタブ切り替え
③ div形式(カード型)店舗ごとにdivカードで箇条書き
④ ショートコード形式[shop_info id="123"]形式
⑤ 非対応・不明「要確認」としてスキップし人手判断へ
ℹ️
検出したブロックは開始タグ〜終了タグ全体をDOM単位で置換し、紹介文などブロック外のテキストは変更しません。置換後は目印コメントを付与し次回実行を高速化します。

差分検知設計

比較ロジック・色付けルール
  • フィールド単位で比較し、全角半角等の表記揺れは正規化して吸収（誤検知防止）
  • 比較対象外: store_id / hpb_url / review_count / rating（頻繁に微変動するためノイズ防止）
  • 情報源の優先順位を維持: info_source_fieldsに記録済みの項目は、再取得時もHPBではなく公式HPを情報源として比較する
  • 実質的な差分があったセルのみ黄色(#FFFF00)にハイライト
  • 差分は変更履歴シートへ追記のみ（削除・上書きしない＝監査性の担保）

よくある質問

クライアントからよく聞かれる質問への回答
Q店舗情報はどこから取得していますか？公式サイトの情報は使わないのですか？

ホットペッパービューティー（HPB）を第一情報源とし、HPBに記載がない項目のみ各店舗の公式ホームページで補完します。HPBに情報がある項目を公式HPの値で上書きすることはありません（詳細は「情報源の優先順位」セクション参照）。

Q公式HPにも情報がない店舗はどうなりますか？

該当項目はnull（空欄）のまま「要確認」として記録され、無理に値を埋めることはありません。担当者が手動で確認・補完する運用を想定しています。

Q記事の紹介文まで自動で書き換わってしまいませんか？

QSlackがないと使えませんか？
